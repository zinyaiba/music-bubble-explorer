# 要件定義書

## はじめに

本ドキュメントは、Music Bubble Explorer V2にライブ情報管理機能を追加するための要件を定義する。この機能により、ユーザーはライブ公演の情報（ツアー、単独公演、フェス）を登録・編集・閲覧できるようになる。また、ナビゲーションの改善として、タグ関連ページの統合も行う。

## 用語集

- **ライブ（Live）**: 音楽公演イベントを表すデータエンティティ
- **ライブ種別（Live_Type）**: ライブの種類（ツアー、単独公演、フェス）
- **セトリ（Setlist）**: ライブで演奏される楽曲のリスト
- **セトリ項目（Setlist_Item）**: セトリ内の1曲を表すエンティティ
- **日替わり曲（Daily_Song）**: ツアー公演で日によって異なる楽曲
- **会場（Venue）**: ライブが開催される場所
- **公演地（Tour_Location）**: ツアーにおける各公演の開催地名
- **ナビゲーション（Navigation）**: アプリ下部のメインメニュー
- **タグページ（Tag_Page）**: タグ一覧とタグ登録を統合したページ
- **livesコレクション**: Firebase Firestoreにおけるライブ情報専用のコレクション

## 要件

### 要件 1: ナビゲーションメニューの変更

**ユーザーストーリー:** ユーザーとして、シンプルで使いやすいナビゲーションを利用したい。メニュー項目を整理することで、目的の機能に素早くアクセスできるようにしたい。

#### 受け入れ基準

1. ナビゲーションは「楽曲」と「タグ」の間に「ライブ」を新しいメニュー項目として表示すること
2. ナビゲーションは「タグ一覧」と「タグ登録」を1つの「タグ」メニュー項目に統合すること
3. ユーザーが「タグ」メニュー項目をタップした場合、システムは統合されたタグページに遷移すること
4. ナビゲーションは「TOP」「楽曲」「ライブ」「タグ」「お知らせ」の5項目を表示すること

### 要件 2: タグページの統合

**ユーザーストーリー:** ユーザーとして、タグ一覧とタグ登録を1つのページで切り替えて使いたい。ページ遷移なしで両方の機能にアクセスできるようにしたい。

#### 受け入れ基準

1. ユーザーがタグページにアクセスした場合、システムは「タグ一覧」と「タグ登録」のタブインターフェースを表示すること
2. ユーザーが「タグ一覧」タブをタップした場合、システムは既存のタグ一覧機能を表示すること
3. ユーザーが「タグ登録」タブをタップした場合、システムは既存のタグ登録機能を表示すること
4. システムはブラウザナビゲーションをサポートするため、選択されたタブの状態をURLに保持すること
5. URLにタブが指定されていない場合、システムはデフォルトで「タグ一覧」タブを表示すること

### 要件 3: ライブデータモデル

**ユーザーストーリー:** 開発者として、ライブ情報を適切に保存・取得できるデータ構造を定義したい。

#### 受け入れ基準

1. ライブデータモデルは以下を含むこと: id、ライブ種別、公演名、会場名、日時、セトリ、および任意の公演地
2. ライブ種別は「tour」（ツアー）、「solo」（単独公演）、「festival」（フェス）のいずれかであること
3. ライブ種別が「tour」の場合、ライブデータモデルは公演地の指定を許可すること
4. セトリはセトリ項目オブジェクトの順序付き配列であること
5. セトリ項目は以下を含むこと: 楽曲ID（任意）、楽曲名、順序、日替わり曲フラグ
6. 日替わり曲フラグがtrueの場合、そのセトリ項目はツアー公演の日替わり曲としてマークされること
7. ライブ情報はFirebase Firestoreの「lives」コレクションに保存すること（songsコレクションとは別）

### 要件 4: ライブ一覧表示

**ユーザーストーリー:** ユーザーとして、登録されたライブ情報を一覧で確認したい。ライブの基本情報を素早く把握できるようにしたい。

#### 受け入れ基準

1. ユーザーがライブ一覧ページにアクセスした場合、システムは登録されたすべてのライブをリスト形式で表示すること
2. ライブ一覧の各項目は、ライブ種別、公演名、会場名、日時を表示すること
3. ユーザーがライブ項目をタップした場合、システムはライブ詳細ページに遷移すること
4. システムはデフォルトで日時の降順（新しい順）でライブをソートすること
5. ライブが登録されていない場合、システムは空の状態メッセージを表示すること

### 要件 5: ライブ詳細表示

**ユーザーストーリー:** ユーザーとして、ライブの詳細情報とセトリを確認したい。日替わり曲がある場合は、それが分かるように表示してほしい。

#### 受け入れ基準

1. ユーザーがライブ詳細を表示した場合、システムはセトリを含むすべてのライブ情報を表示すること
2. セトリ表示は演奏順に楽曲名を表示すること
3. セトリ項目の日替わり曲フラグがtrueの場合、システムは日替わり曲であることを示す視覚的インジケーター（バッジまたはアイコン）を表示すること
4. ライブ種別が「tour」で公演地が指定されている場合、システムは公演地を表示すること
5. セトリ内の楽曲に楽曲IDが設定されている場合、ユーザーがその楽曲をタップすると、システムは楽曲詳細ページに遷移すること

### 要件 6: ライブ登録

**ユーザーストーリー:** ユーザーとして、新しいライブ情報を登録したい。必要な情報を入力してライブを作成できるようにしたい。

#### 受け入れ基準

1. ユーザーがライブ登録ページにアクセスした場合、システムは必要なすべてのフィールドを含むフォームを表示すること
2. ライブ登録フォームは以下を含むこと: ライブ種別セレクター、公演名入力、会場名入力、日時ピッカー、セトリエディター
3. ユーザーがライブ種別として「tour」を選択した場合、システムは追加の公演地入力フィールドを表示すること
4. セトリエディターは、既存の楽曲データベースから楽曲を選択して追加できること
5. セトリエディターは、フリー入力で楽曲名を直接入力して追加できること
6. セトリエディターは、ドラッグ＆ドロップまたは上下ボタンで楽曲の順序を変更できること
7. セトリに楽曲を追加する際、システムは日替わり曲としてマークすることを許可すること
8. ユーザーが有効なフォームを送信した場合、システムはライブをFirebase Firestoreのlivesコレクションに保存すること
9. 必須フィールドが空の場合、システムはバリデーションエラーメッセージを表示すること

### 要件 7: ライブ編集

**ユーザーストーリー:** ユーザーとして、登録済みのライブ情報を編集したい。セトリの変更や日替わり曲の設定を後から修正できるようにしたい。

#### 受け入れ基準

1. ユーザーがライブ編集ページにアクセスした場合、システムは既存のライブデータでフォームを事前入力すること
2. システムはライブ種別、公演名、会場名、日時、公演地、セトリを含むすべてのフィールドの編集を許可すること
3. ユーザーがライブ種別を「tour」から他の種別に変更した場合、システムは公演地フィールドをクリアすること
4. システムはセトリ項目の追加、削除、並び替えを許可すること
5. システムは各セトリ項目の日替わり曲フラグの切り替えを許可すること
6. ユーザーが変更を保存した場合、システムはFirebase Firestoreのlivesコレクション内のライブを更新すること
7. 保存が成功した場合、システムは確認メッセージを表示すること

### 要件 8: ライブ削除

**ユーザーストーリー:** ユーザーとして、不要になったライブ情報を削除したい。誤削除を防ぐための確認機能も欲しい。

#### 受け入れ基準

1. ユーザーがライブの削除を要求した場合、システムは確認ダイアログを表示すること
2. ユーザーが削除を確認した場合、システムはFirebase Firestoreのlivesコレクションからライブを削除すること
3. 削除が成功した場合、システムはライブ一覧ページに遷移すること
4. 削除が失敗した場合、システムはエラーメッセージを表示し、現在のページを維持すること

### 要件 9: ルーティング構造

**ユーザーストーリー:** ユーザーとして、ブラウザの戻る/進むボタンでライブ関連ページを正しくナビゲートしたい。

#### 受け入れ基準

1. システムはライブ一覧ページのルートとして「/lives」を使用すること
2. システムはライブ登録ページのルートとして「/lives/new」を使用すること
3. システムはライブ詳細ページのルートとして「/lives/:liveId」を使用すること
4. システムはライブ編集ページのルートとして「/lives/:liveId/edit」を使用すること
5. システムはタブパラメータ付きの統合タグページのルートとして「/tags」を使用すること
6. システムはすべてのライブ関連ページでブラウザの戻る/進むナビゲーションをサポートすること
